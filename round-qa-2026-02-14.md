# 轮次问答记录（2026-02-14）

## 用户问题
根据这个文档，结合项目代码，请评估下，目前项目的方案选择有问题么？有没有框架重构的必要，还有有什么完善的建议也说下。把每轮我的问题和你的最终答复都写到一个md文件中，放到项目根目录。

## Codex 最终答复
### 1. 总体结论
- 方向总体可行，不建议推倒重来。
- 需要做“架构收敛和模块化重构”，不需要“换技术栈重写”。
- 当前最大问题不在 React/FastAPI 本身，而在方案边界漂移和实现分裂。

### 2. 目前方案选择的主要问题
1. 后端双栈并存，维护成本和决策成本都偏高。  
证据：`server/app.py:27`（FastAPI）与 `server/index.js:1`（Express）同时存在；但运行说明与脚本又以 FastAPI 为主（`README.md:30`、`start_all.bat:23`）。

2. V1 边界有偏移，和需求文档不完全一致。  
证据：需求明确 V1 不做权限细分（`暖通设计管家｜需求说明书（基础版_v_0.md:953`），但代码已有 `admin/user` 角色与全库管理逻辑（`server/app.py:158`、`src/App.tsx:947`）。

3. “记录对、存完整、找得到”目标中的“找得到”支撑不足。  
证据：当前文件上传仅记录文件名和类型到任务版本（`src/App.tsx:411`），没有文件落盘、对象存储、可回查路径或下载接口；只能看到历史文件名，不能真正取回文件内容。

4. 前端单文件过大，后续迭代风险高。  
证据：`src/App.tsx` 约 1096 行，状态、领域逻辑、视图渲染和弹窗流程高度耦合（`src/App.tsx:127` 起）。

5. Tailwind 动态类名策略在生产构建有样式丢失风险。  
证据：大量 `bg-${theme}-...` / `text-${theme}-...` 动态拼接（`src/App.tsx:532`、`src/components/Sidebar.tsx:35`），而 `tailwind.config.js` 未配置 safelist（`tailwind.config.js:1`）。

6. 测试覆盖偏弱，难支撑结构重构。  
证据：当前测试主要是渲染冒烟和占位用例（`src/tests/App.test.tsx:25`、`src/tests/DataHardening.test.tsx:9`）。

### 3. 是否有“框架重构”必要
- 不建议更换 React 或 FastAPI。
- 建议做“同框架内重构”，必要且优先级高：
1. 后端只保留一套（建议 FastAPI），将 Express 版本归档或移除。
2. 前端按领域拆分模块（项目管理、阶段管理、模板与任务、规范资料）。
3. 把模板与任务规则提到独立 domain/service 层，UI 只负责展示和交互。

### 4. 建议的完善路线（按优先级）
1. P0（先做）
- 技术栈收敛：确定 FastAPI 为唯一后端，清理 `server/index.js` 路线。
- 文件记录能力补齐：至少增加文件存储路径、下载查看接口、基础元数据（上传人/时间/来源）。
- 对齐 V1 边界：把“角色权限、主题皮肤、AI相关描述”改为可关闭或移出 V1 主流程。

2. P1（随后）
- `App.tsx` 模块化拆分，补充状态管理边界（可先用 React Context + reducer，不必上重状态框架）。
- 增加接口契约与校验模型（前后端字段统一，减少隐式兜底）。
- 补关键测试：阶段切换、历史只读、完成度计算、模板启停与任务派生。

3. P2（中期）
- 数据层从“单表 JSON blob”逐步拆到可查询结构（项目/子项/阶段/任务/版本）。
- 增加迁移脚本和数据版本号，保证后续 V2 演进可控。

### 5. 一句话判断
项目不需要“换框架重写”，但需要“尽快收敛后端方案 + 按 V1 边界做结构化重构”；否则后续每加一个功能，都会放大维护成本和需求偏移风险。
